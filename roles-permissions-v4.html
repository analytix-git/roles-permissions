Bloco 01 de 05
As sete entidades
Fundamentos

Sete entidades, um mecanismo único de autorização

O modelo v4 separa Permission, Role, Identity, Membership, Tenant e Room como entidades independentes. Roles são definidos por tenant. Permissões são registros reais no banco vinculados a roles via relação M:N (tabela role_permissions).

1
PERMISSION
Ação granular. Cada permissão é um registro no banco (ex: file.create, room.archive) com campo resource_type para agrupamento. Vinculada a roles via tabela role_permissions (M:N). Uma permissão pode pertencer a vários roles sem duplicação.
2
ROLE
Coleção nomeada de permissões. Definido por tenant — cada tenant pode ter roles customizados. Não é mais um enum global fixo. Possui slug (identificador estável para código) e name (label para UI).
3
IDENTITY
Quem a pessoa é. Existe independentemente de qualquer tenant. Tem e-mail único. Pode ter múltiplas memberships em tenants diferentes (1:N).
4
MEMBERSHIP
A relação. Liga Identity a um Tenant com um Role. Constraint: UNIQUE(identity_id, tenant_id) — uma pessoa por tenant, mas pode estar em N tenants. Campos de lifecycle: status (pending/active/suspended/deactivated), invited_at, accepted_at, deactivated_at, invited_by.
5
TENANT
A empresa-cliente. Fronteira de isolamento dos dados. Campo type para categorização.
6
ROOM
Espaço colaborativo dentro do tenant. Relação 1:N com Tenant. Room soma acesso contextual, nunca promove poder acima do tenant.
7
AUDITLOG
Registro imutável de cada ação. Campos: identity_id, action, resource_type, resource_id, tenant_id, timestamp, metadata.

Multi-membership

Identidade pode pertencer a múltiplos tenants — com regras de negócio

A constraint de banco é UNIQUE(identity_id, tenant_id) — uma pessoa não pode ter duas memberships no mesmo tenant, mas pode em tenants diferentes. A restrição single-tenant vira regra de negócio.

1
A REGRA
Single-tenant por padrão. Para a maioria dos usuários, o sistema limita a uma membership ativa. Mas a porta não está trancada no banco — a validação é na aplicação.
2
CONSULTORES
Multi-tenant quando faz sentido. Consultores que atuam em duas empresas podem ter memberships em ambas — com roles diferentes em cada uma (ex: editor na Empresa A, viewer na Empresa B).
3
TRANSFERÊNCIA
Desativa, não deleta. Se uma pessoa sai de uma empresa, membership antiga é desativada (preserva auditoria) e nova pode ser criada no tenant destino.
Bloco 02 de 05
Roles por tenant
Tenants-cliente

Cinco roles padrão — customizáveis por tenant

Cada tenant pode ter seus próprios roles com permissões customizadas. Os cinco abaixo são o template padrão criado quando um tenant é provisionado. O tenant_admin pode, no futuro, criar roles adicionais ou ajustar permissões.

1
TENANT_ADMIN
Gerencia a empresa. Convida membros, define roles, configura integrações, gerencia métodos de autenticação.
2
EDITOR
Cria e edita todo conteúdo. Pode deletar, mudar visibility, vincular a salas. O 'trabalhador de força'.
3
CONTRIBUTOR
Cria, mas só edita o próprio. Role intermediário para colaboradores sem poder sobre obra alheia.
4
VIEWER
Só lê. Não cria, não edita, não comenta fora de salas. Stakeholders e auditores ocasionais. Vê arquivos públicos do tenant.
5
GUEST
Zero visibilidade no tenant. Não vê arquivos públicos, não vê salas, não vê outros membros. Só acessa salas onde foi explicitamente adicionado como room_member ou room_viewer. Caso de uso: médico consultor externo, auditor pontual, parceiro.
Operação da plataforma

O super_admin não é um role do produto — é do deploy

Criar tenants, provisionar infraestrutura, gerenciar billing — tudo isso fica fora do modelo de roles do produto. O super_admin é um papel de operação resolvido no deploy (CLI, painel interno, scripts de provisionamento). Não aparece na matriz, não passa pelo can(), não é um registro na tabela Role.

1
SEPARAÇÃO CLARA
Produto ≠ infraestrutura. O produto resolve autorização de quem usa a plataforma. O deploy resolve quem cria e opera a plataforma. São domínios diferentes com ferramentas diferentes.
2
RESPONSABILIDADES
O que o super_admin faz no deploy: cria tenants, define planos, configura integrações globais, provisiona o primeiro tenant_admin de cada empresa. Nada disso é feature do produto.
3
SEGURANÇA
Superfície de ataque mínima. super_admin nunca é exposto via API pública. Acesso via ferramentas internas com autenticação própria (SSH, painel admin, CI/CD).
Escopo Sala

Três roles de sala — que somam ao tenant, nunca substituem

Ser room_member dá acesso à sala e direito de interagir — mas nunca eleva capacidades acima do role de tenant. Sala adiciona escopo, não promove poder.

1
ROOM_OWNER
Controla a sala. Convida membros, edita settings, arquiva, deleta. Tipicamente quem criou a sala.
2
ROOM_MEMBER
Participa plenamente. Vê arquivos da sala, comenta, reage, adiciona conteúdo. Role-base de quem foi convidado.
3
ROOM_VIEWER
Consome e interage. Comenta, reage, mas não adiciona conteúdo. Stakeholders convidados pontualmente.
Acesso externo

Guest no tenant = porta de entrada para salas, nada mais

Um médico consultor precisa ver os dashboards de uma sala específica, mas não é funcionário da empresa. Ele não deveria ver arquivos públicos, outras salas ou a lista de membros do tenant. O role guest resolve isso: membership real no tenant, mas com zero visibilidade fora das salas onde foi adicionado.

1
O MODELO
Guest tem membership no tenant, mas não vê nada dele. Sem room role atribuído, o guest tem acesso a literalmente zero. O can() nega tudo no escopo tenant para guests.
2
DOIS PASSOS
tenant_admin cria o guest. Depois, room_owner adiciona à sala. Separação clara: quem decide que o externo entra no tenant (admin) ≠ quem decide em qual sala ele participa (owner da sala).
3
VISIBILIDADE
Sala + perfis dos participantes. O guest vê o conteúdo vinculado à sala e os perfis básicos (nome, avatar) de quem está naquela sala. Nada mais.
4
CASO DE USO
Médico consultor, auditor externo, parceiro. Pessoa que precisa de acesso pontual a um espaço específico sem fazer parte da organização. Membership guest pode ser desativada a qualquer momento pelo tenant_admin.

Bloco 04 de 05
Convite & autenticação por tenant
Identidade

Como uma pessoa entra na plataforma — e por que isso depende do tenant

Cada tenant pode ter métodos de login diferentes. Um hospital usa Microsoft SSO; uma consultoria prefere magic link. O convite precisa saber quais portas estão abertas antes de mandar o e-mail.

1
MICROSOFT SSO
Login federado via Entra ID. Ideal para empresas que já centralizam identidade na Microsoft. O usuário nunca cria senha na Nodian.
2
E-MAIL + SENHA
Credencial local clássica. O convidado cria uma senha na Nodian. Exige política de complexidade e rotação futura.
3
E-MAIL + CÓDIGO
OTP enviado ao e-mail. Cada login gera um código de uso único. Sem senha para gerenciar, mas exige acesso ao e-mail a cada sessão.
4
MAGIC LINK
Link de acesso único por e-mail. Um clique autentica. Experiência mais fluida, mas depende da segurança do e-mail do usuário.
Configuração

Métodos de login são feature flags no tenant — ligados ou desligados

Cada tenant carrega uma lista de auth_methods habilitados. Novos métodos podem ser adicionados no futuro sem alterar o modelo — basta registrar um novo tipo e ligá-lo nos tenants desejados.

Regras e cuidados

O que o fluxo de convite precisa garantir

1
TOKEN COM ESCOPO
O link de convite carrega tenant + identity + role. Validade curta (ex: 72h). Expirou? Precisa reenviar. Token nunca expõe dados do tenant no URL — só um ID opaco.
2
MEMBERSHIP PENDING
Convite aceito ≠ membership imediata. A membership nasce como pending e só vira active após o convidado completar a autenticação. Isso evita membros-fantasma.
3
MÉTODO DESABILITADO
Se o tenant desabilitar um método após o convite, o convidado ainda consegue aceitar — desde que reste pelo menos um método ativo. O link resolve métodos no momento do clique, não no momento do envio.
4
REENVIO SEGURO
Reenviar convite invalida o token anterior. Apenas o último token é válido. Evita que links antigos circulem.
5
AUDITORIA
Tudo logado. Quem convidou, quando, qual método foi usado para aceitar, IP, user-agent. O audit trail do convite é tão importante quanto o do acesso.
6
SSO MANDATÓRIO
Tenant pode forçar SSO exclusivo. Se Microsoft SSO é o único método ativo, não há escolha — o convidado é redirecionado direto para o fluxo Microsoft. Isso atende requisitos de compliance corporativo.

Decisões pendentes
Discussão 01 de 03

Quem pode criar ExternalRef quando as empresas tiverem autonomia?

Embedding cruza fronteira de sistemas. Quanto risco vocês aceitam? Hoje ExternalRefs são criados no deploy (super_admin). No futuro, isso pode migrar para o produto — a questão é se fica para qualquer editor ou restrito ao tenant_admin.

1
OPÇÃO A · EDITOR PODE
Mais fluido, mais risco. Qualquer editor adiciona Metabase à sala. Multiplica vetores de embedding e exige curadoria da allowlist.
2
OPÇÃO B · APENAS TENANT_ADMIN ★
Fricção que vale a pena. Embedding como ato administrativo. Pedir ao admin é leve atrito, mas evita expor sistemas sem critério.
Discussão 02 de 03

A allowlist de origens é global ou por tenant?

Cada ExternalRef aponta para uma origem. Permitir tudo é inseguro; bloquear tudo é inútil. Quem mantém a lista?

1
OPÇÃO A · GLOBAL (DEPLOY) ★
Curadoria central. super_admin mantém origens aprovadas via deploy. tenant_admin escolhe quais usar. Simples e seguro.
2
OPÇÃO B · POR TENANT
Autonomia total. Cada empresa gerencia suas origens. Mais flexível, mas exige que admin entenda risco de cada domínio.
Discussão 03 de 03

Há revisão obrigatória antes de um ExternalRef ficar visível?

ExternalRef tem problema de 'fronteira herdada': se o Metabase tem dashboards não-públicos e embedamos como visibility=tenant, podemos expor o que o sistema externo nunca exporia.

1
OPÇÃO A · SEM REVISÃO
Confiança no criador. Quem cria é responsável. Mais ágil, mas engano fica visível imediatamente.
2
OPÇÃO B · OPERACIONAL ★
super_admin aprova manualmente via deploy. Nos primeiros meses, processo manual. Aprende com casos reais antes de automatizar.
3
OPÇÃO C · WORKFLOW
Estado pending_review no produto. Mais pesado, mas auditável e escalável para o futuro.
